# 設計資料ファイルインデックス命名規則

## 目的

設計資料に一意のインデックスを付与し、人間がファイルを素早く見つけられるようにする。

## インデックス形式

```
[A-Z][0000-9999]
```

- **先頭1文字**: アルファベット大文字（A-Z）— カテゴリを表す
- **後続4桁**: 数字（0001-9999）— カテゴリ内の連番

例: `A0001`, `B0012`, `C0003`

## カテゴリ定義

| プレフィックス | カテゴリ | 説明 |
|---|---|---|
| A | Architecture | システム全体のアーキテクチャ設計 |
| B | Backend | バックエンド設計（API、サーバー、DB） |
| C | Component | コンポーネント設計・共通部品 |
| D | Data | データモデル・スキーマ設計 |
| E | External | 外部連携・API連携設計 |
| F | Frontend | フロントエンド画面設計・UI/UX |
| G | General | その他の汎用的な設計資料 |
| S | Specification | 機能仕様書 |
| T | Test | テスト設計・テスト計画 |

※ 必要に応じてカテゴリは追加可能。

## ファイル命名規則

```
{インデックス}_{kebab-case-の説明}.md
```

例:
- `A0001_system-architecture.md`
- `B0001_user-api-design.md`
- `F0001_login-screen-design.md`
- `S0001_user-registration-spec.md`

## 適用ルール

1. **設計資料を新規作成する際は、必ずインデックスを付与すること。**
2. **インデックスは `docs/index.md` の台帳で管理する。** 新規作成時は台帳に追記する。
3. **同一インデックスの重複は禁止。** 台帳を確認してから採番する。
4. **各カテゴリ内で連番とする。** 欠番は許容する（削除されたドキュメントの番号は再利用しない）。
5. **設計資料の格納先は `docs/` ディレクトリ配下とする。**

## 台帳ファイル

台帳は `docs/index.md` に置く。形式は以下の通り:

```markdown
| インデックス | ファイル名 | 概要 | 作成日 |
|---|---|---|---|
| A0001 | A0001_system-architecture.md | システム全体構成図 | 2026-01-15 |
```

## ワークフロー

1. 設計資料を作成する必要が生じる
2. `docs/index.md` を確認し、該当カテゴリの最新番号を確認する
3. 次の連番でインデックスを採番する
4. ファイルを `docs/{インデックス}_{説明}.md` として作成する
5. `docs/index.md` の台帳にエントリを追記する
